Week 1 · How AI Actually Thinks Deck slides 1–3 · Prompt as a question
Core concept
Old computers matched exact words. Modern AI understands meaning. A neuron's only job is to fire (ping) when enough neighbours fire first. Learning = adjusting vote strengths, not rewriting rules.
The analogy that lands
Old search = robot librarian who only fetches a book if you say the exact title. Say "hound" instead of "dog" — nothing. AI = a friend who knows you want a dog book even if you say "furry animal that fetches sticks."
Activity first — no laptop (15 min)
Mind Reader Game: JK describes an animal's vibe only. You guess. Swap. "Loves chasing spherical objects, sleeps on my feet" = dog.

Ping Handshake: Hold hands in a circle. Squeeze = Ping. Brain only raises other hand if two squeezes arrive simultaneously. You are a neural network.
What to say when stuck
"But HOW does it understand?" → "That's next week. Short answer: millions of tiny on/off switches that vote."

"Is it actually thinking?" → "Scientists disagree. Hinton says our brains do the same with neurons — which makes it hard to say AI is just pretending."
Tech introduced this week
• HTML — a language that describes structure. Browser reads it, builds the page.
• Browser — turns HTML into what you see.
• API — a door that lets two programs talk. Waiter analogy: you (app) order via waiter (API), kitchen (Claude) makes it.
Prompting exercise
Ask Claude the same question 5 times, each richer:

1. "what is AI"
2. "What is artificial intelligence?"
3. "Explain AI to a 10-year-old"
4. "Explain AI to a 10-year-old who loves cricket, using cricket analogies"
5. "I'm 10, love cricket and robots. Explain how AI thinks using things I already know"

Write: what changed between 1 and 5? Which was most useful? Why?
🟢 Zero-tech parent — just copy this
No API, no Replit, no setup needed this week. Open Claude.ai together and paste:
Explain how AI works to my 10-year-old son who loves cricket and building robots. Use analogies from cricket. Keep it under 200 words.
Then ask JK: "Does that feel right? What would you change about the explanation?"
Build progress
Nothing built yet — intentional. This week builds the mental model that makes everything else make sense.
Robotics connection
The PJ touch sensor IS a neuron — fires when pressure crosses a threshold, exactly like the Ping Handshake.

Week 2 · Talking to AI Like a Pro Deck slides 4–5 · Prompt as a specification
Core concept
AI converts words into features and predicts meaning. "Tuesday" and "Wednesday" fire almost identical neuron patterns. The Super-Guesser picks the statistically most likely next word from billions of examples.
The analogy that lands
Phone autocomplete trained on every book ever written. "Peanut butter and..." → "jelly" (99%). Not smart — just seen that sequence billions of times. The 1986 breakthrough: let the computer fix its own trillion connections by working backwards from mistakes. Like adjusting your cricket bat after a miss — but every muscle simultaneously.
Activity first — no laptop (10 min)
One-Word Story: Each person adds one word. "The... cat... sat... on..." Notice your brain autocompletes. Then give Claude the same 4 words. Compare. Is Claude more logical? More surprising?
What to say when stuck
"Is AI just guessing?" → "Yes — from having read everything humans ever wrote. Your brain guesses too — 'I'll be...' → you think 'back' before they finish."
Tech introduced this week
• JavaScript — makes pages interactive. HTML = walls of a house. JS = lights and switches.
• Function — a named set of instructions. "whenButtonPressed() → open camera"
• Camera API — a door built into every browser that lets JS ask for camera access.
Prompting exercise
Ask Claude: "Build me a single HTML page with one big button in the centre. When tapped, ask for camera permission and show the live camera view. Nothing else."

Save the output. Open in Safari. Does it work? If not — don't fix it. Ask JK: "What did Claude misunderstand? How would you describe the problem back to it?" Let him iterate. This broken page = Prototype Zero.
🟢 Zero-tech parent — just copy this
If Replit feels overwhelming, just open Claude.ai and paste this for JK:
Build me a single HTML page. It has one large button that says "Show Camera". When I tap it, it asks for camera permission and shows the camera view below the button. Make the button big and easy to tap on a phone. Give me the complete HTML file.
Save the response as a .html file on the laptop. Open it in Safari. Done.
Build progress
End of week: one HTML page, one button, camera opens. No AI yet. That's the foundation.

Week 3 · First Working Prototype Deck slide 5 · Prompt as a build brief
Core concept
The Synthesis Ladder — how AI sees images. Pixel → edge → beak → bird. Trillions of these chains fire simultaneously in 0.01 seconds. This is what happens when your app sends a photo to Claude.
The analogy that lands
Open Google Images. Upload any animal photo. Watch it identify instantly. Say: "Every layer of the Synthesis Ladder just fired to produce that." Then: "When our app sends a photo to Claude, the same thing happens — and Claude describes what it saw in words."
Activity first (5 min)
Show Deck Slide 5. Walk the ladder: pixel → edge → beak → bird. Ask: "If you were a neuron on Level 2, what's your only job?" (Answer: detect one specific edge.) That specificity = how it gets so accurate.
What to say when stuck
"Why does it get things wrong?" → "Bad lighting changes the edges. The pattern doesn't match. It guesses wrong. That's why we add a honesty note in Week 5."
Tech introduced this week
• API key — secret password that identifies your app to Claude. Never share it.
• JSON — structured info between programs. Like a labelled lunchbox: {food: "sandwich", filling: "cheese"}.
• Base64 — how images become text to travel through the internet.
Prompting exercise
JK prompts Claude (include Prototype Zero code in the prompt):

"Here is my existing HTML camera page. Add a 'Describe' button. When tapped: capture a camera frame, convert it to base64, send it to the Claude API using this key [KEY], and show the description on screen."

The key realisation JK needs to make himself: Claude needs the existing code as context. Don't hint — let him discover this.

Here is my HTML camera page: [PASTE JK'S CODE HERE] Add a big "Describe" button. When tapped, capture a frame from the camera, convert it to base64, send it to the Claude API (key: [PASTE KEY]) and display what Claude says it sees. Keep it simple.
Build progress
End of week: camera → Claude API → text description. The first time the app does something genuinely useful. Celebrate this.

Week 4 · Design for a Real User Deck slides 6–7 · Prompt as a design brief
Core concept
RLHF — without human feedback AI just predicts text, including toxic text. Humans score outputs. Scores adjust connection strengths. The person writing the reward rules has enormous power. This week JK writes his own reward rules: the system prompt.
The analogy that lands
Training a puppy. Treat = reward signal. "No" = penalty. Different trainers → different personalities. Point at JK's system prompt: "You are the trainer this week."
Activity first (10 min, no laptop)
Hot & Cold: Hide an object. JK is the AI. You can only say "beep" (warm) or "bop" (cold). Watch how fast he adapts from feedback alone. That IS RLHF.

Persuasion Detective: Convince JK to eat broccoli. Round 1: facts. Round 2: connect to his favourite cricket player. Notice how Round 2 feels different — that's what bad actors exploit.
What to say when stuck
"Who decides what's a good AI answer?" → "Whoever writes the reward function. At Anthropic, hundreds of human raters. In JK's app — JK."
Tech introduced this week
• System prompt — hidden instructions that shape Claude's entire behaviour.
• Web Speech API — browser's built-in voice. One line does it.
• PWA — Safari → Share → Add to Home Screen → real app icon.
Prompting exercise
Part 1 — System prompt: JK prompts Claude: "Help me write a system prompt for an AI that describes images to blind users. The tone should be [JK decides]. Always mention [JK decides]. Never [JK decides]."

Part 2 — UI: "Update my app so it's easy to use one-handed with eyes closed. Big button. High contrast. Speak the description aloud automatically."

Test: close eyes. Can you use the app? What needs to change?

Write a system prompt for an AI assistant that describes images to blind users. It should: be warm and friendly, always describe colours and textures, mention if something looks dangerous, never assume what the person wants to do with the object.
Paste Claude's output into the system prompt field in the app code. JK can edit it however he likes.
Build progress
End of week: app speaks, has a personality JK wrote, installs on iPhone. For the first time it feels like a real app.

Week 5 · Safety, Guardrails & Trust Deck slide 6 · Prompt as a safety spec
Core concept
AI hallucinates — it confidently describes things that aren't there. For a sighted person: annoying. For a blind person trusting your app: genuinely dangerous. This week: build an app that knows when it might be wrong and says so.
The analogy that lands
A sensor can be fooled — put a mirror in front of an object sensor and it reads nothing. AI can be fooled too. The Prism Prank from the deck: reporting what sensors detected isn't lying — it's a confident wrong answer. Hallucination is the same.
Activity (10 min)
Spot the Fake: 3 AI images from Midjourney + 3 real news photos. Mixed up. Can you tell? What gives them away? Connect: if the image was fake, what else might be fake about the story?
What to say when stuck
"Should we not build it then?" → "An honest imperfect description beats silence. The key word is honest — that's what we build this week."
Tech introduced this week
• Error handling — what happens when API fails or camera refuses
• Loading state — spinner while waiting
• Graceful degradation — app stays useful even when something breaks
Prompting exercise
Part 1 — Honesty: Update system prompt: "If uncertain about any detail, say so. Use 'I think I can see...' or 'this might be...' rather than stating everything with equal confidence."

Part 2 — Errors: Add loading spinner, camera-fail message, API-error fallback.

Stress test: Dark room. Blurry object. Hand covering lens. Does the app handle these honestly?
🟢 Zero-tech parent — just copy this
Update my image description app's system prompt to add: if the AI is uncertain about any detail, it should say "I'm not certain, but this might be..." rather than stating things confidently. Also add a polite note if the image is too dark or blurry to describe well.
Build progress
End of week: app is functionally complete. Works, speaks, honest about uncertainty, fails gracefully. Week 6 is polish only.

Week 6 · Polish, Install & Showcase Deck slides 8–9 · Prompt as a pitch
Core concept
AI can be backed up and revived forever — but it can't feel what it's like to hand something useful to someone who needed it. That feeling drove JK through 5 weeks of frustration. AI can't be motivated by that. Humans can. That's the superpower.
Closing activity (Deck Slide 9)
Human Superpower Map: Draw robot + human. List 3 things robot does better by 2030. List 3 things only humans can do. Which list excites JK more?

Read the Family Manifesto aloud together. Then ask: "Which of our 6 sessions surprised you most?" Let JK answer fully before you share yours.
What to say when stuck
"What if someone asks how it works at school?" → "You built it. You can explain every decision you made. That's more than most adults can say about the AI they use daily."
Tech introduced this week
• PWA install: Safari → Share → Add to Home Screen → real icon, full-screen
• manifest.json — gives the app its name, icon colour, and installability
• Hosting: Replit gives a public URL anyone can visit
Prompting exercise
Part 1 — Install: Ask Claude to add a manifest.json so the app installs on iPhone. JK chooses the name and icon colour. Install it. Open from home screen. That moment = the payoff.

Part 2 — Showcase prep: "I'm 10, presenting this app at school. Give me 10 tough judge questions and help me write honest answers."

The demo: Blindfold a family member. Hand them the phone. One tap. Phone speaks. Practise until reliable 8/10 times.
